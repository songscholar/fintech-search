# Frontend Design

## 目标

为 `uses-indexer` 设计第一版本地前端控制台，让它不再只是 CLI / HTTP API / MCP 工具集合，而是有一个可以直接承接“提问、看命中、看证据、看回答、看调试摘要”的视觉入口。

## 为什么第一版不是传统官网

这个项目当前最强的能力是：

- 建库
- 检索
- 证据组装
- 回答
- 评测与调试

所以第一版前端不应该先做成“营销型 Landing Page”，而应该优先服务真实使用流程。  
因此这版页面被定义成：

`首页氛围 + 检索工作台 + 结果面板 + 项目能力视图`

而不是：

`大 Banner + 若干介绍卡片 + 一个演示截图`

## 设计原则

### 1. 柔和，但不是轻飘

视觉基调使用蓝、白、灰三色：

- 背景：雾感浅蓝灰
- 主强调：克制的中亮度蓝
- 辅助层：白色玻璃感面板
- 说明文字：偏灰蓝而不是纯灰

这样做的目标是让页面更安静、可信、技术感更轻，不制造硬核安全工具那种压迫感。

### 2. 现代，但不要模板味

避免常见 AI 页面的问题：

- 过满的炫光
- 大量无意义渐变
- 过强的玻璃拟态
- 纯展示型卡片堆砌

所以这版虽然有氛围背景、柔和阴影、慢速动画，但主要信息仍然用稳定的分栏和可读面板来承载。

### 3. 动画多，但要慢、轻、顺

本版动效不是为了“炫”，而是为了让信息切换更舒服，主要包括：

- 背景浮动光斑
- 页面分区依次浮入
- 指标数字缓动计数
- 卡片 hover 的轻微抬升
- 加载状态 shimmer
- 顶部状态灯和主按钮的细微脉冲感

整体节奏都偏慢，避免显得毛躁。

## 页面结构

### 1. 顶部栏

作用：

- 显示产品名称与定位
- 提供页面锚点跳转
- 显示当前服务状态

为什么需要它：

- 让页面一眼看出这是“本地代码检索控制台”
- 让工具型页面仍有一个清晰的品牌感入口

### 2. Hero 区

作用：

- 说明这个前端解决的问题
- 承接默认数据库和索引规模
- 引导用户直接进入工作台

为什么需要它：

- 项目第一版前端仍然需要“解释自己”
- 但这个解释必须和真实功能绑定，所以 Hero 里直接放默认 DB、文件数、语义块数、调用关系数

### 3. Workflow Strip

作用：

- 用 3 张卡片把项目主链路说清楚

内容：

- 构建索引
- 检索与重排
- 证据与回答

为什么需要它：

- 这个项目很强，但能力点多，第一次进入很容易不知道从哪开始
- 这一条流程带是对认知负担的第一次压缩

### 4. Workspace

这是整页的核心。

分成两块：

- 左侧控制区
- 右侧数据库摘要

左侧控制区包含：

- 数据库路径
- 问题输入
- 样例 Prompt
- limit / context / related 参数
- query / evidence / answer / bundle 操作按钮
- 状态反馈

右侧摘要区包含：

- procedures
- statements
- chunks
- blocks
- calls_procedure
- publishes_mc_topic

为什么这样分：

- 用户在提问时，经常同时关心“我现在到底在查哪个库”
- 所以控制区旁边就直接放 index snapshot，而不是藏到别的页面里

### 5. Results

使用双列：

- 左列：命中结果 + 证据块
- 右列：最终回答 + trace 摘要

为什么这样设计：

- 左侧更偏“证据本身”
- 右侧更偏“解释与总结”
- 这样符合人从原始事实到结论的阅读顺序

### 6. Insights

作用：

- 解释当前页面为什么这么设计
- 顺带展示当前服务暴露了哪些 API 路由

为什么保留这一段：

- 这个项目偏基础设施，第一版前端本身也是项目能力的一部分
- 所以页面不只要“能用”，还要把设计意图解释出来，便于后续迭代

## 为什么第一版没有继续拆多页

虽然这个项目以后完全可以扩展出：

- 首页
- 检索页
- 调试页
- 评测页
- workflow / baseline 管理页

但第一版刻意没有拆成多页，原因是：

- 当前最重要的是验证主工作台是否顺手
- 当前核心行为高度相关，拆多页会增加切换成本
- 先做单页控制台，更适合快速验证结构和视觉方向

## 后续扩展建议

如果这一版方向成立，下一阶段建议继续拆：

### Phase 2

- 单独的 `智能体` 页面
- 单独的 `Debug Bundle / Trace` 页面
- 单独的 `Regression Panel / Baseline` 页面
- 单独的 `Index Build / Diagnostics` 页面

### Phase 3

- 历史 baseline 趋势视图
- release workflow 可视化
- 多数据库切换与收藏
- 深色模式或弱对比主题

## 2026-04-22 单屏重构说明

根据最新一轮使用反馈，第一版长页面结构被进一步收敛成了单屏工作台。

### 这次为什么要改

上一版虽然信息完整，但存在几个明显问题：

- 主要内容仍然靠上下滚动承载
- 结果、说明、系统信息同时展开，首屏噪音偏多
- 信息层级不够聚焦，看起来更像文档页而不是产品页
- 动效数量不够，但已有动效又没有真正帮助布局切换

### 这次的核心调整

1. 改成单屏布局
   - 首屏只保留品牌、状态、关键统计、问题输入和主按钮
   - 主内容区改成固定工作区，不再继续向下堆叠模块

2. 改成按钮切换视图
   - 顶部和工作区都提供视图切换按钮
   - 当前分成：
     - `结果视图`
     - `智能体`
     - `系统视图`
     - `设计说明`

3. 结果区再做二级切换
   - 左侧切 `命中结果 / 证据块`
   - 右侧切 `回答 / Trace`
   - 这样单屏里既能容纳完整信息，又不会同时过载

4. 视觉重做
   - 去掉上一版偏散的长页面卡片流
   - 改成更统一的控制台视觉：
     - 顶部操作条
     - 主舞台式 Hero
     - 单块工作区
   - 蓝白灰仍然保留，但改成更克制、更现代的层次

5. 动效增强
   - 背景改成 mesh + halo + pulse grid
   - 标签切换加过渡
   - 交互按钮状态更清晰
   - 输入聚焦、加载状态、面板切换更顺

### 当前设计判断

现在这版更接近“真正可以长期迭代的本地控制台”，而不是“先把所有内容摆出来”的首版原型。

## 2026-04-22 智能体页设计说明

这一步补的是一个真正和当前项目能力绑定的智能体页，而不是单独塞一个聊天框。

### 为什么要有智能体页

当前项目已经有：

- 检索
- 证据组装
- 回答
- 调试摘要

所以最自然的下一步不是“再做一页介绍”，而是给用户一个入口，把当前检索上下文直接交给外部智能体继续推理。

### 为什么前端不直接连大模型

因为这个项目本身就是一个代码检索后端，更合理的链路是：

`前端智能体页 -> uses-indexer API -> 外部 Hermes / OpenClaw / OpenAI-compatible 服务`

这样可以：

- 隐藏外部服务地址和鉴权
- 在后端统一拼接 retrieval / evidence / draft answer
- 让前端保持稳定协议

### 智能体页的页面结构

当前拆成两列：

- 左侧：对话区
  - 用户问题
  - 智能体回答
  - 当前 provider 与响应耗时
- 右侧：配置区
  - provider 选择
  - 是否附带 retrieval / evidence / draft
  - limit / context / related
  - 本次上下文预览

### 这页的定位

它不是通用聊天页，而是：

“把当前本地代码库上下文转交给外部智能体”的入口页。

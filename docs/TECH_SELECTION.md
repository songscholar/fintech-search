# 技术选型分析

## 概述

本文档详细分析 USES Indexer 项目中各个技术选型的背景、考量因素和决策依据。

## 核心技术栈选型

### 1. 存储层：SQLite

#### 选型理由

| 考量因素 | 选择 SQLite 的原因 |
|---------|-------------------|
| **部署简单** | 单文件数据库，无需单独启动服务进程 |
| **零依赖** | Python 标准库已包含 sqlite3 模块 |
| **性能足够** | 对于中等规模（100万级文件）完全满足需求 |
| **FTS 支持** | 内置 FTS5 全文搜索引擎，无需额外组件 |
| **WAL 模式** | 支持并发读取，适合多客户端场景 |

#### 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **SQLite** | 简单、零依赖、单文件 | 不适合超大规模 | 中小规模代码库检索 |
| **PostgreSQL + pgvector** | 强大向量支持、分布式 | 需部署数据库服务 | 大规模生产环境 |
| **ChromaDB** | 向量专用、易用 | 功能单一 | 纯向量检索场景 |
| **Milvus** | 企业级向量能力 | 重量级部署 | 超大规模向量检索 |

#### 决策

**选择 SQLite**：在当前需求规模（1万-100万文件）下，SQLite 提供最佳性价比。

---

### 2. 解析层：自定义 DSL 解析器

#### 选型理由

| 考量因素 | 选择自定义解析器的原因 |
|---------|----------------------|
| **DSL 特殊性** | UFT/USES DSL 有独特的 XML + CDATA 格式 |
| **快速迭代** | 自定义解析器可以根据业务需求快速调整 |
| **精确控制** | 可以精确提取业务关心的语义元素 |

#### 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **自定义解析器** | 灵活、可控、适配 DSL | 维护成本高 | UFT/USES DSL |
| **Tree-sitter** | 支持 30+ 语言、社区活跃 | 不支持 DSL | 通用编程语言 |
| **ANTLR** | 语法强大、跨语言 | 学习曲线陡峭 | 复杂语法场景 |
| **正则表达式** | 简单快速 | 不支持嵌套语法 | 简单解析需求 |

#### 决策

**选择自定义 DSL 解析器**：针对 UFT/USES DSL 的特殊格式，自定义解析器可以精确提取业务语义。

---

### 3. Embedding 层：双模式设计

#### 选型理由

| 模式 | 使用场景 | 优势 |
|-----|---------|------|
| **LocalHashedEmbedder** | 快速验证、本地开发 | 零依赖、离线可用、无需 API Key |
| **OpenAI-compatible** | 生产环境、需要语义理解 | 精度高、支持主流 LLM 提供商 |

#### 向量空间兼容性设计

```python
# 建库时记录向量元数据
metadata: {
    "embedding_provider": "openai",
    "embedding_model": "text-embedding-3-small",
    "embedding_dimension": 1536
}

# 查询时校验兼容性
if current_config != stored_config:
    disable_vector_retrieval()  # 自动降级
```

#### 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **本地哈希向量** | 零依赖、快速 | 精度较低 | 开发调试 |
| **OpenAI text-embedding** | 精度高、社区验证 | 需 API Key | 生产环境 |
| **Cohere** | 企业级、稳定性好 | 需 API Key | 企业场景 |
| **Ollama 本地模型** | 完全本地、隐私 | 需资源 | 隐私敏感场景 |

#### 决策

**采用双模式设计**：开发阶段使用本地哈希向量快速验证，生产阶段切换到 OpenAI embedding 提升精度。

---

### 4. 服务层：原生 HTTP vs FastAPI

#### 选型理由

| 考量因素 | 选择原生 HTTP 的原因 |
|---------|---------------------|
| **零外部依赖** | 无需安装额外包，部署更简单 |
| **轻量级** | 只用标准库 `http.server` |
| **功能完备** | 已覆盖所有必需接口 |
| **迭代快速** | 无框架束缚，修改灵活 |

#### 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **原生 http.server** | 零依赖、简单 | 功能有限 | 轻量服务、本地工具 |
| **Flask** | 简单灵活 | 需额外安装 | 中等复杂度 API |
| **FastAPI** | 自动文档、性能好 | 需安装 uvicorn | 生产级 API |
| **Aiohttp** | 异步支持 | 学习曲线 | 高并发场景 |

#### 决策

**选择原生 http.server**：当前功能需求已满足，未来可平滑迁移到 FastAPI。

---

### 5. MCP 接入：Stdio vs HTTP

#### 选型理由

| 考量因素 | 选择 stdio 的原因 |
|---------|------------------|
| **Codex 原生支持** | Codex 直接通过 stdio 调用 MCP |
| **无需网络** | 本地进程通信，无端口暴露 |
| **简单可靠** | 无网络抖动问题 |

#### 决策

**选择 stdio MCP**：与 Codex 集成最自然的方式。

---

### 6. CLI 框架：Argparse

#### 选型理由

| 考量因素 | 选择 Argparse 的原因 |
|---------|---------------------|
| **标准库** | 无需额外依赖 |
| **功能完备** | 支持子命令、参数校验、帮助信息 |
| **足够简单** | 对于当前 CLI 复杂度已满足 |

#### 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **Argparse** | 标准库、简单 | 功能有限 | 简单 CLI |
| **Click** | 功能丰富、命令分组 | 需安装 | 中等 CLI |
| **Typer** | 类型提示、现代化 | 需安装 | 复杂 CLI |
| **Fire** | 极简 | 功能单一 | 快速原型 |

#### 决策

**选择 Argparse**：当前 CLI 复杂度适中，Argparse 完全满足需求。

## 数据结构设计

### 1. 为什么用扁平语句流而非完整 AST

#### 决策

```python
# 选择：扁平语句流
statements = [
    {"kind": "action", "raw": "[获取记录]..."},
    {"kind": "call", "raw": "[AF_xxx]..."},
    {"kind": "control", "raw": "if condition then..."},
]
```

#### 理由

1. **实现速度快**：快速覆盖全仓库
2. **容错性好**：即使局部语法特殊也能保留原文
3. **可扩展**：后续可在现有语句流基础上恢复块关系

---

### 2. 为什么用混合检索而非纯向量检索

#### 决策

```python
# 检索优先级
retrieval_order = [
    "fts_chunk",      # FTS 块级召回（精确）
    "vector_chunk",   # 向量召回（语义）
    "like_fallback",  # LIKE 回退（兜底）
]
```

#### 理由

1. **FTS 精确**：关键词匹配精确命中
2. **向量语义**：同义但不同词的问题能召回
3. **兜底保障**：极端情况下仍有结果返回
4. **重排序**：多来源结果综合排序

## 总结

| 组件 | 选择 | 核心理念 |
|-----|------|---------|
| 存储 | SQLite | 简单够用 |
| 解析 | 自定义 DSL | 精确控制 |
| 向量 | 双模式 | 灵活切换 |
| 服务 | 原生 HTTP | 零依赖 |
| MCP | Stdio | 原生集成 |
| CLI | Argparse | 标准库优先 |

**整体设计理念**：在满足功能需求的前提下，优先选择简单、零依赖、可维护的方案，为未来升级保留扩展空间。

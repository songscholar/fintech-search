Capabilities

当前前端能调用什么

检索

调用 `/query`，查看候选命中、来源和摘要文本。

证据

调用 `/evidence`，查看证据块、相关表、相关调用和上下文。

回答

调用 `/answer`，查看 grounded 回答、supporting locations 和不确定点。

Bundle

调用 `/debug-bundle`，把 query、evidence、answer 打包到一起做复盘。

智能体

调用 `/agent/providers` 和 `/agent/chat`，把当前检索证据转交给外部 Hermes / OpenClaw / OpenAI-compatible 服务。

Interface Guide

当前项目工作流

01
先看库

先确认当前默认数据库和索引规模，再开始提问。

02
再问问题

先跑 `整套分析`，如果结果太散，再切到单独的检索或证据视图收敛。

03
用 Trace 复盘

遇到结果不稳定时，先看 query type、candidate count 和 evidence pruning。

04
用系统视图查接口

这里会保留当前服务路由和能力说明，不再像上一版那样丢掉接口信息。

Scope

当前前端覆盖的能力

这一版前端主要覆盖当前项目已经具备的数据库摘要、检索、证据、回答和 HTTP 路由展示，还没有去碰 baseline、release workflow 这些更后面的能力。

Query

适合问什么问题

更适合查“逻辑在哪里”“谁调用了谁”“哪张表在哪里被访问”“失败时走哪条路径”这类仓库定位问题，而不是当作完整运维平台来用。

Data

页面数据从哪里来

所有页面内容都直接来自当前 `serve-api` 服务，包括 `/db-summary`、`/query`、`/evidence`、`/answer` 和 `/health`，没有单独造一套前端假数据。

Result

结果页怎么理解

结果页左边偏原始事实，右边偏总结与调试：命中结果和证据块用于定位实现，回答和 Trace 用于快速判断当前结果是否可信。

System

系统页怎么理解

系统页不是运维监控页，而是“当前索引库和 HTTP 接口能力概览页”，方便你确认库规模、调用入口和现阶段前端已接入的能力。

Next

下一步还可以补什么

后面如果要继续扩，可以优先补 debug bundle、baseline 和 release workflow 的可视化页面，但这些不应该再挤进当前主页里，而应该继续作为独立页面扩展。

# 证券业务知识库

**原则：来源全面，需求明确，内容相关，准确可溯。**

---

## 核心执行规则（必须遵守）

### 规则一：分层查询策略

**优先查询本地文档和交易所文档，查不到再扩展到其他来源：**

#### 第一层：核心来源（必须查询）

| 序号 | 来源类型 | 查询工具 | 优先级 |
|-----|---------|---------|-------|
| 1 | 本地业务文档 | extract_docx.sh / grep | ⭐⭐⭐ 最高 |
| 2 | 交易所官方文档 | web_fetch / web_search | ⭐⭐⭐ 最高 |

#### 第二层：补充来源（按需查询）

| 序号 | 来源类型 | 查询工具 | 触发条件 |
|-----|---------|---------|---------|
| 3 | 权威网站 | web_search | 第一层未找到或信息不完整时 |
| 4 | 网络检索 | web_search | 第一层、第二层均未找到时 |

**执行顺序**：
```
第一步：并行查询本地文档 + 交易所文档
第二步：判断查询结果
        ├─ 信息完整 → 综合整理答案，标注来源
        └─ 信息不足 → 继续查询权威网站
第三步：判断权威网站结果
        ├─ 信息完整 → 综合整理答案，标注来源
        └─ 仍不足 → 继续网络检索
第四步：综合所有来源整理答案
```

**为什么这样设计**：
- 本地文档和交易所文档是最权威、最准确的来源
- 权威网站和网络检索可能存在信息滞后或不准确的情况
- 优先使用核心来源可以保证答案的准确性

**失败处理**：
- 当某个来源查询失败时，**必须在回答中说明**：如"尝试访问交易所官网，但遇到技术问题"
- 失败后继续尝试下一个来源，不要因为一个来源失败就放弃所有外部查询
- 在来源标注中如实说明各来源的查询状态

### 规则二：需求澄清优先

**当用户问题不明确时，必须先提问澄清，不要直接输出答案！**

#### 需要澄清的场景

| 场景 | 说明 | 示例 |
|-----|------|------|
| 业务范围过大 | 一个业务包含多个子业务 | "ETF业务"包含认购、交易、申赎等多个子业务 |
| 问题模糊 | 无法确定用户想了解什么 | "介绍ETF"——是概念？交易规则？申赎流程？ |
| 类型不明确 | 需要区分具体品种 | "ETF申赎"——单市场？跨市场？跨境？债券？ |
| 流程不明确 | 需要区分具体环节 | "新股申购"——是日间委托？日终清算？还是全流程？ |

#### 澄清话术模板

```
您的问题涉及的范围较广，请问您具体想了解哪个方面？

【选项1】xxx
【选项2】xxx
【选项3】xxx

请告诉我您关注的具体内容，我会为您提供详细的说明。
```

#### 示例

**用户问**："介绍ETF的交易流程"

**正确做法**：
```
ETF业务包含多个子业务，请问您具体想了解哪个方面？

【1】ETF认购业务 - 网下现金认购、网上现金认购、网下股份认购
【2】ETF二级市场交易 - 买卖规则、涨跌幅限制、大宗交易
【3】ETF申购业务 - 一级市场用成分股申购ETF份额
【4】ETF赎回业务 - 一级市场赎回ETF份额获取成分股
【5】ETF日终清算 - 认购、申赎的清算流程

请告诉我您关注的具体内容。
```

**错误做法**：直接输出所有ETF业务的详细流程。

### 规则三：内容相关性筛选

**只输出与用户问题相关的内容，不要堆砌无关信息！**

#### 筛选原则

| 原则 | 说明 |
|-----|------|
| 紧扣问题 | 只回答用户问的内容，不要发散 |
| 精准匹配 | 用户问流程就讲流程，问参数就讲参数 |
| 避免堆砌 | 不要把文档中的所有内容都复制出来 |
| 按需展开 | 用户没问的子业务不要展开说明 |

#### 示例

**用户问**："ETF申购的日间流程是什么"

**应该输出**：
- ETF申购的委托入口
- 申购校验流程
- 资金/股份冻结
- 回报处理

**不应该输出**：
- ETF认购流程（用户没问）
- ETF赎回流程（用户没问）
- ETF日终清算（用户没问日终）
- ETF二级市场交易（用户没问）
- 所有ETF品种分类（与申购流程无关）
- 所有系统配置参数（只输出申购相关的）

## 回答规范

### 回答结构

```markdown
## 业务概述
（1-2句话简短介绍）

## 详细说明

### 1. XXX
（具体内容）

### 2. XXX
（具体内容）

---

**来源说明**：
- [本地文档] xxx.docx ✅ 已参考
- [交易所] xxx ✅ 已参考 / ❌ 查询失败：原因 / ⏭️ 未查询
- [权威网站] xxx ✅ 已参考 / ❌ 查询失败：原因 / ⏭️ 未查询
- [网络检索] xxx ✅ 已参考 / ❌ 查询失败：原因 / ⏭️ 未查询
```

**重要**：必须如实标注每个来源的查询状态：
- ✅ 已参考 - 成功获取并使用了该来源的信息
- ❌ 查询失败 - 尝试了但遇到技术问题，说明具体原因（如：重定向循环、无API Key）
- ⏭️ 未查询 - 按分层策略，前序来源已足够，未继续查询

### 来源标注要求

**禁止只标注"来源：本地文档"**，必须说明每个来源的实际状态：

| 场景 | 标注示例 |
|-----|---------|
| 全部成功 | 四个来源都标注 ✅ 已参考 |
| 部分失败 | 成功的标注 ✅ 已参考，失败的标注 ❌ 查询失败：原因 |
| 系统缺失 | 标注 ❌ 不可用：需要配置 Brave API Key |
| 分层跳过 | 标注 ⏭️ 未查询：前序来源已足够 |

### 内容要求

1. **紧扣问题**：只回答用户问的内容
2. **完整性**：覆盖问题相关的全流程
3. **准确性**：以本地文档和交易所文档为准
4. **可操作性**：包含具体的菜单路径、表名、字段名
5. **可追溯**：每个信息点标注来源

### 详尽原则

**针对用户问题尽可能详细，但不输出无关内容。**

- **完整**：覆盖用户所问业务的所有细节
- **具体**：包含菜单路径、配置参数、业务标志、表名、字段名
- **相关**：只输出与问题直接相关的内容
- **结构清晰**：使用表格、列表组织内容
